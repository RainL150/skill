# Concept Research - 把概念搞明白

你遇到一个新词/新概念，想搞清楚它到底是啥、怎么用、跟别的有啥区别——这个 skill 就是干这个的。

**核心思路**：不只是搜一下就完事，而是多个来源对比验证，最后整理成一个靠谱的笔记。

## 什么时候用

- 看到一个新术语，想快速搞懂（比如「GRPO 是啥」）
- 写文章/做 PPT 前，想先把概念捋清楚
- 技术选型时，想对比几个方案的区别
- 学新东西时，想要一个靠谱的总结

## 举个例子

**场景 1**：你在看 DeepSeek 的论文，看到 GRPO 这个词，不知道啥意思

```
你：研究一下 GRPO
我：
  1. 搜一圈（原始论文、技术博客、教程）
  2. 对比各个来源说的是否一致
  3. 整理成一个结构化笔记：定义、原理、跟 PPO 的区别、什么时候用
```

**场景 2**：你想搞懂 WebSocket 和 SSE 的区别

```
你：WebSocket vs SSE 有什么区别
我：
  1. 分别搜两个概念
  2. 找对比文章
  3. 整理成对比表：性能、适用场景、实现复杂度
```

**场景 3**：调研一个市场/领域

```
你：帮我调研一下 AI Agent 这个领域
我：
  1. 搜定义和边界
  2. 搜主要玩家和产品
  3. 搜最新动态
  4. 整理成调研报告
```

## 我怎么工作

### Step 1：先搞清楚你问的是啥

拿到一个概念后，我会先判断：

- **技术概念**（编程、框架、算法）→ 去官方文档、GitHub、技术博客找
- **学术概念**（论文术语）→ 去 arXiv、Google Scholar 找原始论文
- **商业概念**（市场、产品）→ 去新闻、行业报告找
- **通用概念**→ Wikipedia 通常够用

然后生成几个搜索变体。比如你问「MoE」，我会搜：
- MoE
- Mixture of Experts
- 混合专家模型
- MoE 原理
- what is MoE

### Step 2：多个来源搜一圈

不是只搜一个链接就完事。我会：

1. **先搜官方/权威的**（论文、官方文档）
2. **再搜解释性的**（技术博客、教程）
3. **最后搜问答类的**（Stack Overflow、知乎）

每个来源我会标记可信度：
- 🟢 **高**：原始论文、官方文档、知名研究者博客
- 🟡 **中**：技术博客、教程、百科
- 🔴 **低**：随便一个博客、问答帖子

**抓取工具选择**：

| 工具 | 优先级 | 说明 |
|-----|-------|------|
| Firecrawl | 默认首选 | 输出更干净，成功率更高 |
| WebFetch | 备选 | 没有 Firecrawl API Key 时使用 |

Firecrawl 用法（需要配置 `FIRECRAWL_API_KEY`）：
```bash
# 抓取单个页面
curl -X POST 'https://api.firecrawl.dev/v2/scrape' \
  -H 'Authorization: Bearer $FIRECRAWL_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{"url": "目标URL", "formats": ["markdown"]}'
```

如果没有配置 `FIRECRAWL_API_KEY`，回退到 WebFetch。

### Step 3：交叉验证（这是重点）

这是跟普通搜索的区别。我会对比各个来源：

**如果大家说的一样** → ✓ 置信度高，直接用

**如果只有一个来源提到** → ⚠️ 标记「存疑」，建议你再确认

**如果来源之间矛盾** → ❌ 标记「冲突」，把两边说法都列出来，不替你做决定

举个例子，假设我在调研 GRPO：

```
定义：
  - 来源 A（论文）：一种 RL 算法，用组相对优势估计
  - 来源 B（博客）：一种 RL 算法，用组相对优势估计
  - 来源 C（教程）：一种 RL 算法，用组相对优势估计
  → ✓ 3/3 一致，置信度 0.95

历史：
  - 只有来源 A（论文）提到：2024.02 首次提出
  → ⚠️ 单源，置信度 0.6，建议确认

应用场景：
  - 来源 A：主要用于 LLM 推理训练
  - 来源 D：广泛用于推荐系统
  → ❌ 冲突！两边说法不同，需要你判断
```

### Step 4：整理成笔记

最后输出一个结构化的 Markdown 文件。

**重要：解释风格要求**

不要只堆专业术语和公式。每个重要概念/公式后面，都要加上**通俗解释**或**具体例子**帮助理解：

```markdown
## 定义

{正式定义}

> **通俗理解**：用类比或大白话再解释一遍，让没背景的人也能懂

## 原理

{正式的原理说明，包括公式}

> **举个例子**：用具体场景或数字说明这个原理是怎么工作的

比如讲 GRPO 的优势计算，不要只写公式，要加上：
> 比如一道题 64 个答案，40 个对（奖励=1），24 个错（奖励=0），
> 平均奖励=0.625。那答对的优势是 +0.375，答错的是 -0.625。

## 应用场景

{正式说明}

> **实际案例**：某某公司/项目用这个技术做了什么，效果如何

## 对比

| 维度 | A | B |
|-----|---|---|

> **一句话总结**：A 适合 XX 场景，B 适合 YY 场景

## 来源
- [论文链接] - 可信度高
- [博客链接] - 可信度中

## 置信度
哪些是确定的，哪些是存疑的
```

**风格原则**：
- 专业内容要保留（公式、术语、精确定义）
- 但每个专业内容后面跟一个 `> 通俗理解/举例` 块
- 用类比、数字例子、实际场景来解释抽象概念
- 目标：让一个聪明但没这个领域背景的人也能看懂

## 两种模式

### 快速模式（默认）

搜 3-5 个核心来源，快速出结果。适合：
- 快速了解一个新词
- 不需要太深入

### 深度模式 (--deep)

搜 10+ 来源，覆盖更全。会额外包括：
- 优缺点分析
- 局限性和边界
- 实现细节/代码示例
- 最新进展
- 实际案例

适合：
- 技术选型
- 写正式文档
- 深度调研

## 几个注意事项

### 什么情况效果好

- 有明确定义的技术概念（算法、框架、协议）
- 有原始论文或官方文档的概念
- 有多个来源讨论的概念

### 什么情况效果一般

- 太新的概念（可能搜不到足够来源）
- 太宽泛的概念（比如「AI」本身太大了，建议细化）
- 主观性强的概念（比如「最好的编程语言」）

### 遇到问题怎么办

| 情况 | 处理 |
|-----|------|
| 搜不到结果 | 换个关键词，或者问你确认概念名称 |
| 来源太少 | 标记「置信度低」，建议你补充来源 |
| 来源严重冲突 | 暂停，让你选择信任哪个 |
| 概念太宽泛 | 建议你细化，比如「AI」→「Transformer」|

## 输出格式

默认输出到 `research/concepts/{概念名}.md`

文件结构：

```
---
concept: 概念名
confidence: 0.85  # 整体置信度
sources_count: 6  # 来源数量
updated: 日期
---

## 定义

{正式定义，一两句话}

> **通俗理解**：{用类比或大白话解释}

## 原理

{正式原理，包括公式（如有）}

> **举个例子**：{用具体数字或场景说明}

## 应用场景

{主要用途}

> **实际案例**：{具体的应用实例}

## 对比（如有）

| 维度 | A | B |
|-----|---|---|
| ... | ... | ... |

> **一句话总结**：{什么时候用 A，什么时候用 B}

## 来源
| 来源 | 可信度 | 链接 |
|-----|-------|-----|
| 原始论文 | 🟢 | ... |
| 技术博客 | 🟡 | ... |

## 置信度报告
| 维度 | 状态 | 说明 |
|-----|------|-----|
| 定义 | ✓ 一致 | 5/5 来源相同 |
| 历史 | ⚠️ 存疑 | 只有 1 个来源 |
```

**关键**：每个专业内容后面都要有 `> 通俗理解/举例` 块，不能只堆术语。

## 跟其他工具的区别

| 工具 | 干什么 | 什么时候用 |
|-----|-------|----------|
| **concept-research** | 搞懂一个概念 | 想要靠谱的、验证过的知识 |
| **perplexity-research** | 快速问答 | 简单问题，不需要深入 |
| **WebSearch** | 原始搜索 | 只是想找几个链接 |

核心区别：concept-research 会**交叉验证**，不只是搜一下就完事。
