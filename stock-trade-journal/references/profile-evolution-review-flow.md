# 交易画像更新复盘流程

用于用户要求“更新画像”“深化画像”“根据最近交易修正画像”“复盘单只股票并沉淀画像证据”“重新生成交易画像”等场景。

画像更新不是改几句描述。每次更新都必须是一场完整复盘：用交易结果、持仓变化、笔记、外部数据和纪律执行情况判断哪些习惯值得保留，哪些习惯必须被拦截。

## 1. 两种复盘范围

### 单标的画像复盘

用于复盘某一只股票或某一笔交易链条：

```bash
python3 scripts/profile_review.py single <代码> --json --write-docs
```

必须回答：

- 原 `trade_decision` 是什么？
- 后续 `holding_review` 是否改变了主逻辑？
- 当前动作是否符合 profile 拦截器？
- 这只标的暴露出什么可复制好习惯或应规避坏习惯？
- 这只标的的发现是否需要进入下一次全记录画像更新。

单标的复盘不得直接更新整体画像或正式 profile。除非用户明确要求“把这条单标的发现强行写入 profile”，也必须先提示这是单一样本、证据等级不足，并建议等待全记录复盘确认。

### 全记录画像更新复盘

用于更新整个交易画像：

```bash
python3 scripts/profile_review.py all --json --write-docs
```

必须回答：

- 哪些交易行为在多次样本中重复出现？
- 哪些是有结果验证的优势，哪些只是 note 里写得合理？
- 哪些风险偏差反复出现？
- 当前 profile 是否仍适用，还是需要新增/拆分 profile？
- 应保留、加强、删除或新增哪些交易拦截器？

## 2. 默认生成两份文档

每次画像复盘或画像更新，都必须默认生成两份 Markdown。两份文档的职责不能混在一起。

全记录画像更新输出：

1. 自读画像文档：给用户看，用来理解自己的交易习惯、风格、优势、风险、漂移和验证缺口。
2. 可执行 profile 草案：给 `profiles/` 侧使用，用来沉淀交易前拦截器、提醒卡、规则和更新依据。

默认路径：

```text
~/.trade-journal/results/trade-journal/profile-reviews/<日期>-self-portrait.md
~/.trade-journal/results/trade-journal/profile-reviews/self-portrait-latest.md
~/.trade-journal/results/trade-journal/profile-reviews/<日期>-executable-profile-draft.md
~/.trade-journal/results/trade-journal/profile-reviews/executable-profile-draft-latest.md
```

单标的复盘只输出局部复盘，不更新整体画像；第二份文档是 profile 证据包，不是可执行 profile 草案。它只能作为后续全记录画像更新的输入。

```text
~/.trade-journal/results/trade-journal/profile-reviews/<日期>-<代码>-self-portrait.md
~/.trade-journal/results/trade-journal/profile-reviews/<日期>-<代码>-profile-evidence.md
~/.trade-journal/results/trade-journal/profile-reviews/single-profile-evidence-latest.md
```

旧兼容入口 `profile-summary-latest.md` 可以继续同步，但它只能是自读画像的别名，不代表 profiles 侧草案。

自读画像至少包含：

- 一句话画像。
- 样本概览。
- 交易周期偏好。
- 买入触发方式。
- 建仓与加仓习惯。
- 卖出与止损习惯。
- 持仓集中度。
- 标的选择范围。
- Note 验证：哪些只是当时想法，哪些被证据支持。
- 已验证/可复制好习惯。
- 潜在优势，但必须继续验证。
- 需要规避或调整的坏习惯。
- 交易拦截器摘要。
- 当前审计发现。
- 交易质量审计。
- 适合的 profile。
- 为什么不是其他 profile。
- Profile 草案与下一步。
- 交易前提醒。
- 待验证问题。

全记录可执行 profile 草案至少包含：

- Note 验证门槛。
- 可复制好习惯候选。
- 待验证假设。
- 需要规避的行为与拦截器依据。
- 现有拦截器。
- 审计发现到拦截器变更。
- 建议写入 profile 的固定规则。
- 画像更新记录模板。

单标的 profile 证据包至少包含：

- 该标的的 note 验证门槛。
- 该标的暴露出的局部待验证假设。
- 该标的触发的拦截器或纪律缺口。
- 是否建议进入下一次全记录画像更新。
- 明确声明：不单独更新整体画像，不单独写入全局 profile。

## 3. Note 审计验证

`note` 是当时的交易主张，不是验证后的事实。每次复盘必须先审计 note，再判断好坏习惯。

逐条 `trade_decision` 至少检查：

- 是否包含主逻辑、交易周期、失效条件、加仓/再买规则。
- 是否有关联交易和后续持仓变化。
- 是否已有卖出闭环，可以检查结果与卖出纪律。
- 是否已有后续 `holding_review`，以及是否发生主逻辑或交易周期切换。
- 是否需要外部验证：行情走势、财报/经营指标、公告/事件进展、监管变化、商品供需/天气/库存等。

`profile_review.py` 的输出只是本地审计和外部验证队列，不是最终画像结论。凡是 `verification_next_step` 显示“可立即查外部数据验证”的记录，复盘时必须继续查外部数据，不能把它们简单留在“待验证”。

审计结论只能落入以下几类：

| 状态 | 含义 | 画像处理 |
|------|------|----------|
| `closed_loop_available` | 已有卖出/交易闭环 | 可结合收益和退出纪律评 A 级 |
| `reviewed_open_position` | 未清仓但已有持仓复盘 | 可评 B 或 B/C，仍需外部数据 |
| `unverified_note_only` | 只有当时 note | 先查外部数据；查不到或仍无闭环时才写待验证假设 |
| `incomplete_note` | 字段缺失 | 不得作为画像规则依据 |

没有经过这一步，不允许写“复制好习惯”“规避坏习惯”“新增强拦截器”。

## 3.1 外部验证要求

全记录画像更新必须完成外部验证，除非数据源不可得。可查的就必须查，不允许只停留在本地 note 审计。

按资产类型验证：

- 个股成长逻辑：查最新财报、收入/利润/现金流、核心经营指标、管理层指引。
- 题材/事件逻辑：查公告、新闻、事件进展、事件后价格反应。
- 商品/ETF 逻辑：查底层商品价格、库存、供需、天气、官方周报或基金净值。
- 技术/低吸逻辑：查买入后价格路径、回撤、是否突破/跌破原计划区间。
- 已清仓交易：查卖出后价格路径，判断是纪律兑现、过早卖出还是卖在情绪高点。

外部验证结果必须把样本分成：

| 结果 | 含义 | profile 处理 |
|------|------|--------------|
| `validated` | 外部数据支持原 note，且没有明显反证 | 可升级或保留为 A/B 级证据 |
| `partially_validated` | 有支持也有风险，仍需跟踪 | 只能写成有条件规则 |
| `contradicted` | 外部数据明显反证原 note | 写入坏习惯或拦截器依据 |
| `inconclusive` | 外部数据不足或无法确认 | 保留为待验证，不写规则 |

最终自读画像和可执行草案必须同时展示：

- 本地 note 审计结果。
- 外部验证结果。
- 哪些原待验证样本已被升级、降级或反证。
- 哪些仍必须等待未来 holding_review 或卖出闭环。

## 4. 证据等级

好坏习惯不能只凭主观感受写入画像。每条结论都必须标注证据等级：

| 等级 | 含义 |
|------|------|
| `A` | 已完成买卖闭环，结果和纪律执行可验证 |
| `B` | 未清仓但已有外部数据、价格行为或多次复盘支持 |
| `C` | 只有交易/复盘笔记支持，尚未被结果验证 |
| `D` | 只是推断或用户自我描述，不得写成画像规则 |

只有 `A` 和强 `B` 可以写入“可复制好习惯”或“必须规避坏习惯”。`C` 只能写成待验证假设。`D` 不进入画像。

## 5. 好习惯写法

好习惯必须可复制，不能只写“有耐心”“有逻辑”。

好习惯必须来自已验证证据，不能来自“note 写得好”。如果只是当时笔记完整，但没有卖出闭环、后续复盘或外部数据验证，只能写入“潜在优势，待验证”。

推荐格式：

```text
好习惯：
- 行为模式：
- 适用条件：
- 复用步骤：
- 禁用场景：
- 证据：
- 证据等级：
```

例：

```text
等待回撤后小仓试错
- 适用条件：基本面未恶化，价格回撤来自情绪或估值压缩。
- 复用步骤：先 1/4 仓，等经营数据或价格确认后再加。
- 禁用场景：财报转弱、监管/商业模式破坏、只是错过买点后追涨。
```

## 6. 坏习惯写法

坏习惯必须转成交易拦截器，否则没有约束力。

坏习惯也必须有证据。字段缺失、集中度过高、策略切换缺少复盘这类结构性纪律问题可以直接形成拦截；但“某种买入逻辑不好”必须先经过行情、财报或结果验证。

推荐格式：

```text
坏习惯：
- 行为模式：
- 触发条件：
- 风险后果：
- 拦截器：
- 例外条件：
- 证据：
- 证据等级：
```

例：

```text
错过买点后追加强势
- 触发条件：价格已明显高于原计划买点，且没有新基本面证据。
- 拦截器：只能小仓确认，禁止一次性补足目标仓位。
```

## 7. 画像拦截器更新规则

每次画像更新必须输出：

```text
保留的拦截器：
加强的拦截器：
新增的拦截器：
删除或降级的拦截器：
待验证假设：
```

新增或加强拦截器必须说明：

- 触发条件。
- 要拦截的错误动作。
- 允许的替代动作。
- 例外条件。
- 证据来源。
- 证据等级。
- 是否来自 note 审计、交易闭环、持仓复盘、外部数据，还是仅为待验证提醒。

## 8. Profile 文件更新要求

写入 profile 前必须先展示完整复盘结论。只有用户明确确认“写入/保存/更新 profile”时，才修改 `profiles/<slug>.md`。

Profile 文件必须保留这些章节：

```text
适用场景
禁用场景
可复制好习惯
需要规避的坏习惯
交易拦截器
交易前提醒卡
持仓复盘规则
画像更新记录
```

画像更新记录必须写明：

```text
日期：
复盘范围：单标的 / 全记录
更新依据：note 审计 / 交易闭环 / 持仓复盘 / 外部数据 / 风险偏差 / 纪律缺口
保留：
新增：
加强：
降级：
待验证：
```

## 9. 输出顺序

每次画像复盘默认输出：

1. 复盘范围和数据口径。
2. 原画像规则摘要。
3. Note 审计验证结果。
4. 交易质量审计。
5. 已验证/可复制好习惯。
6. 潜在优势和待验证问题。
7. 需要规避的坏习惯。
8. 拦截器变更建议。
9. 是否建议更新 profile。
10. 已生成的两份 Markdown 路径；单标的是自读画像 + profile 证据包，全记录才是自读画像 + 可执行 profile 草案。
11. 若用户要求写入，给出将修改的 profile 路径和具体 diff 摘要。
