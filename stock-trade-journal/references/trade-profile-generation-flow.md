# 交易 Profile 生成流程

用于用户要求“根据我的交易记录生成 profile”“生成交易画像”“把我的策略沉淀成 profile”“新增一个交易 profile”等场景。

## 目标

生成 profile 的目标不是给用户贴标签，而是把可重复使用的录入约束沉淀下来：

- 适合什么交易形态
- 录入时必须问哪些问题
- 哪些问题只作为可选项
- 如何写入 `record_trade.py --note`
- 如何防止逻辑漂移、摊平错误或策略误切换
- 如何区分“用户有交易解释”和“交易解释真的被结果验证”
- 如何用反证式审计暴露用户可能存在的交易问题，而不是只做正向画像

## 输入数据

优先读取本地数据：

```bash
sqlite3 ~/.trade-journal/results/trade-journal/db/trades.db '.tables'
sqlite3 ~/.trade-journal/results/trade-journal/db/trades.db "select * from trades order by timestamp;"
sqlite3 ~/.trade-journal/results/trade-journal/db/trades.db "select * from positions;"
sqlite3 ~/.trade-journal/results/trade-journal/db/trades.db "select * from watchlist;"
sqlite3 ~/.trade-journal/results/trade-journal/db/trades.db "select * from notes order by timestamp;"
```

如需分析收益归因、回撤、买点质量，再使用行情/财务数据能力补充；不要为了生成 profile 默认联网。

## 生成步骤

1. **先画像，不先写文件**
   - 总结交易频率、持仓周期、买卖比例、分批行为、止盈/止损行为。
   - 总结关注笔记里的关键词，例如“回落、长跌、触底、突破、催化、业绩”。
   - 判断是已有 profile 可覆盖，还是需要新建 profile。
   - 画像结论必须先完整输出，不能只给一个 profile 名称。
   - 画像必须区分两类证据：交易 note 只能证明“当时或事后有解释”，成交结果、退出纪律、仓位变化和后续行情才可能证明“解释有一定有效性”。

2. **识别交易形态**
   - 左侧机会型：下跌、错杀、估值修复、分批低吸。
   - 右侧突破型：突破、放量、趋势确认、业绩持续上修。
   - 周期反转型：供需、价格、库存、开工率、利用率反转。
   - 事件催化型：财报、回购、政策、产品、航线、并购、分红。
   - 核心长期型：护城河、现金流、长期复利、仓位上限。
   - 短线/波段型：明确目标价、止损价、时间窗口。

3. **生成 profile 草案**
   - 不要把所有形态塞进一个 profile。
   - 每个 profile 只覆盖一种主交易形态。
   - 一只股票可以有多逻辑；一笔交易只能指定一个主逻辑。
   - 必填问题保持轻量，默认 3-5 个。

4. **反证式质量审计**
   - 生成画像时必须主动回答“如何确定用户不是错的、不是事后合理化、不是只是运气好”。
   - 对已清仓交易，检查买入逻辑、卖出纪律、实现收益、卖出后是否大幅踏空或回撤。
   - 对未清仓持仓，检查当前浮盈亏、集中度、同一逻辑暴露、是否用“长期看好”兜底短期买点错误。
   - 对多次加仓，检查加仓是新证据增强，还是仅仅因为价格更低、仓位没够或“原逻辑不变”。
   - 对交易 note，标明哪些是用户原始事实，哪些是推断，哪些还需要行情、财报或后续结果验证。
   - 如果数据不足，明确说“目前不能证明有 edge”，不要把画像写成肯定用户没有问题。

5. **让用户确认落盘**
   - 如果用户只是问“我适合什么 profile”，先输出草案。
   - 如果用户明确说“保存/创建/写入 profile”，再写入 `profiles/<slug>.md`。
   - 文件名使用英文小写短横线，例如 `cyclical-bottom.md`。

## 推荐 profile 结构

```markdown
# 交易录入 Profile：<名称>

用于“<适用交易形态>”。

## 触发方式

只有用户明确要求使用该 profile 时才启用。默认 `/stj 记录` 不启用。

## 必填问题

1. 主逻辑：
2. 交易周期：
3. 失效条件：
4. 加仓/再买规则：

## 可选问题

1. 辅助逻辑：
2. 仓位上限：
3. 目标价/观察价：
4. 复盘日期：

## 写入格式

```text
<原始备注>；主逻辑: <答案>；周期: <答案>；失效条件: <答案>；加仓规则: <答案>
```

## 质量要求

- 必填问题不要超过 5 个。
- 失效条件必须可观察。
- 策略切换必须写切换笔记。
```

## 输出要求

生成 profile 建议时，必须输出：

1. 画像结论：用户更像哪种交易形态，必须按下方“画像结论模板”展开。
2. 交易质量审计：必须回答“这些画像结论是否真的被交易结果验证，还是只是 note 里的解释”。
3. 主要问题与反证：列出最可能的行为偏差、失败路径和需要被证明的假设。
4. 为什么不是其他 profile。
5. 建议新建还是复用现有 profile。
6. profile 草案。
7. 是否需要写入文件，以及写入路径。

## 画像结论模板

画像结论要像一篇交易行为复盘，不要像表格字段填空。默认采用“判断型叙述”结构，标题可以保留，但每段必须先给判断，再给数据证据。

### 小标题骨架

完整画像总结只约束小标题，不约束每段具体内容。默认使用以下通用行为维度小标题，允许根据数据略作合并、删减或改写，但整体风格应接近这个骨架：

1. **一句话画像**
   - 用一句话概括用户交易风格，并把最贴近的风格相似性自然融进去，让开头有辨识度。
   - 默认优先给出 1-3 个知名投资者、交易者或交易流派类比，例如彼得·林奇、约翰·邓普顿、乔治·索罗斯、德鲁肯米勒、巴菲特、格雷厄姆等；类比对象必须来自交易数据，不要为了显得生动而硬套名人。
   - 可以写成“你更像……，带一点……”这类判断句；类比对象可以是投资大师、交易流派或行为风格，但必须解释相似点来自哪些行为证据。
   - 名人类比必须加限定：这只是交易行为、选股偏好或风控习惯的相似，不代表收益能力、研究深度、资金规模或完整投资风格等同。
   - 如果数据不足以支持知名人物类比，明确说“暂无足够证据匹配具体人物”，再使用流派或行为风格类比。
   - 不要强制给比例，不要套用固定表述，也不要把类比写成独立清单。

2. **交易周期偏好**
   - 判断偏短线、中线、中长线、长线。
   - 依据交易频率、持仓周期、买卖比例和复盘记录。

3. **买入触发方式**
   - 判断是左侧低吸、右侧突破、事件催化、基本面验证、估值修复、周期反转，还是其他触发。
   - 用交易笔记、关注记录和买入时点说明。

4. **建仓与加仓习惯**
   - 判断是一笔打满、试仓、分批建仓、确认后加仓，还是越跌越买。
   - 用同一标的多次买入记录说明。

5. **卖出与止损习惯**
   - 判断是否会止盈、是否有止损、是否会清仓、是否有策略切换记录。
   - 用已卖出交易、`stop_loss`、`take_profit`、交易备注证明。

6. **持仓集中度**
   - 判断组合是分散、适度集中，还是单票/单行业/单市场/单币种过重。
   - 用仓位占比、币种内集中度、前三大仓位说明。

7. **标的选择范围**
   - 判断覆盖哪些市场、行业和资产类型，例如 A 股、港股、美股、ETF、商品等。
   - 说明宽范围选择是优势还是需要能力圈约束。

8. **主要优势**
   - 总结用户已经做对的行为，例如不频繁交易、会等待、会分批、能止盈。
   - 优势必须由交易行为或结果支撑；如果只是 note 里写得合理但没有成交结果验证，必须降级为“潜在优势”。

9. **主要风险**
    - 总结最需要 profile 约束的行为偏差，例如主逻辑漂移、亏损摊平、仓位过度集中、短线失败改长线。
    - 风险必须包含反证视角：哪些成功可能只是行情 beta、题材运气或事后补 note；哪些失败可能被“长期看好”“原逻辑不变”掩盖。

10. **交易质量审计**
    - 不只判断风格，还要判断交易质量：买点质量、加仓质量、退出质量、仓位质量、失效条件硬度和结果归因。
    - 明确写出“不能证明用户没问题”的地方，例如卖出样本太少、未清仓持仓尚未经历压力测试、note 是事后补充、同一逻辑暴露过高。
    - 对已清仓交易给出实现结果和退出质量判断；对未清仓交易说明尚需用哪些财报、行情或风险事件验证。

11. **适合的 profile**
    - 判断复用现有 profile 还是新建 profile，并说明原因。

12. **profile 草案与下一步**
   - 如果用户只是询问，输出草案并等待确认。
   - 如果用户明确要求保存，再写入 profile 文件。

### 文风要求

- 不要输出机械 checklist；小标题可以固定，段落内容必须自然叙述。
- 多用“你不像……”“你更像……”“这说明……”“真正的问题是……”这种判断句。
- 必须敢于指出用户可能的问题；画像不是夸奖，也不是替用户证明自己正确。
- 不要把“有逻辑”写成“逻辑有效”。必须说明哪些结论已被交易结果验证，哪些只是待验证假设。
- 数据要服务判断，不要堆数字。
- profile 建议放在画像之后，不要一上来就说用哪个 profile。
