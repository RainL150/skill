# 交易前画像拦截流程

用于用户咨询“能不能买”“要不要加仓”“要不要卖”“继续持有吗”等具体交易动作时。

这个流程必须和当前交易画像强绑定。画像不是背景说明，而是交易前的拦截器集合；先判断动作是否被画像允许，再给操作建议。

## 1. 读取上下文

先读取本地标的上下文：

```bash
python3 scripts/analyze_holdings.py context <代码或名称> --json
python3 scripts/profile_review.py single <代码> --json
```

然后读取当前适用 profile：

```text
profiles/<当前画像>.md
```

Profile 选择规则沿用主流程：用户指定优先；如果用户说“我的画像”且只有一个 profile，可自动使用；多个 profile 未指定时必须让用户指定。

## 2. 查外部数据

交易咨询属于当前决策，必须补充最新可核验数据：

- 当前价格、涨跌幅、成交状态。
- 影响主逻辑的最新财报、公告、行业或商品数据。
- 如果是加仓，必须看“新证据”是否增强，而不是只看价格。
- 如果是卖出/减仓，必须区分止盈、风控、逻辑失效和仓位再平衡。

## 3. 先触发画像拦截器

从 profile 的“交易拦截器”章节读取规则。每次交易咨询都必须输出：

```text
画像匹配：符合 / 部分符合 / 不符合
触发的拦截器：
允许动作：
禁止动作：
必须补充的问题：
```

如果触发高优先级拦截器，不能直接给“可以买/可以加仓”。必须先提醒用户该动作违反哪条画像纪律。

## 4. 多轮追问规则

只追问会改变交易结论的问题，最多 3 个。不要问泛泛的“你怎么看”。

### 买入/开仓

必须确认：

- 这是左侧试仓、右侧确认、事件催化，还是追涨补错过？
- 新开仓的首笔仓位是多少，为什么不是更大？
- 错了看哪个指标或价格位？

### 加仓

必须确认：

- 这次加仓的新证据是什么？
- 是计划内分批、基本面增强、突破确认，还是亏损摊平/错过后追？
- 加仓后是否超过 profile 的单票、同逻辑或同市场暴露上限？

### 继续持有

必须确认：

- 原 `trade_decision` 是否仍成立？
- 是否发生策略切换？
- 如果发生切换，是否已有 `holding_review` 写明新逻辑、新失效条件和复盘期限？

### 卖出/减仓

必须确认：

- 是止盈、止损、逻辑失效、仓位过高，还是情绪性卖出？
- 如果卖出后继续看好，重新买入条件是什么？
- 如果不卖，违反了哪条风控或复盘要求？

## 5. 输出交易前提醒卡

最终输出必须先给提醒卡，再给建议：

```text
交易前提醒卡
- 当前动作：
- 当前画像：
- 画像匹配：
- 触发的拦截器：
- 外部数据验证：
- 允许动作：
- 禁止动作：
- 如果仍要交易，必须写入的 note_type：
```

如果信息不足，不要假装确定。直接给出需要补充的问题或数据。

## 6. 写入笔记要求

- 真实交易：写入 `trade_decision`。
- 未交易观察：写入 `watch_observation`。
- 已持仓复盘或策略切换：写入 `holding_review`。

策略切换必须写清：

```text
原逻辑为什么失效；新主逻辑；新失效条件；新复盘日期。
```

如果用户拒绝补充关键字段，仍可给建议，但必须标注“未通过画像纪律确认”。
